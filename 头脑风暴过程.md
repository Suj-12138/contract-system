## 方案选择阶段（第一轮）

### 三种架构方案对比

| 方案 | 架构 | 说明 |
|------|------|------|
| 1 | FastAPI + 原生 HTML/JS MPA + JSON 文件 | 复用度最高，与工单系统一致 |
| 2 | FastAPI + htmx 增强 + JSON | 交互更好，开发量稍多 |
| 3 | FastAPI + Vue 3 SPA + SQLite | 前后端分离，现代架构 |

**第一轮选择：** 方案 3 — 前后端分离

---

## 设计确认过程（方案 3 阶段）

依次确认了 5 段设计内容：

1. **整体架构与目录结构** — 确认（Vue 3 SPA + FastAPI + SQLite）
2. **5 人分工方案** — 确认（P1 认证+基础设施，P2 合同业务，P3 审批引擎+消息，P4 模板+管理员，P5 前端 SPA）
3. **数据库模型**（6 张表） — 确认
4. **API 接口设计**（20 个端点） — 确认
5. **前端页面与路由**（11 个 SPA 页面） — 确认

## 架构转向：从方案 3 回到方案 1

### Q12：转向方案 1

**背景：** 用户提出"想改成方案 1"，并询问分工如何合并（方案 3 的 P5 前端 SPA 角色不再需要）

**问：** 分工合并方案？（A 4人全栈+1人测试 / B 按技术层拆分 / C 5人全栈细粒度）

**答：** A — 4 人全栈开发 + 1 人测试/集成/文档

**方案 1 更新后的分工：**

| 编号 | 分工 | 内容 |
|------|------|------|
| P1 | 认证 + 基础设施 | 注册/登录/会话、JSON 存储层、项目骨架、登录页面 |
| P2 | 合同业务（全栈） | 合同 CRUD + 状态机 + 全文检索 + 合同页面 |
| P3 | 审批引擎 + 消息（全栈） | 两级审批链 + 站内消息 + 到期提醒 + 审批/消息页面 |
| P4 | 模板 + 管理员功能（全栈） | 模板 CRUD + 用户管理 + 数据看板 + 管理页面 |
| P5 | 测试 + 集成 + 文档 + **演示** | API 测试、端到端验证、文档、汇报 PPT、**周五演示主讲** |

---

### Q13：代码合并方式？

**问：** 不同组员写的代码怎么合并？

**答：** 用 Git。设计文档中补充了文件归属表——每人只改自己目录下的文件，几乎不会产生合并冲突。

**后续讨论：** 用户表示组员不太会用 Git，讨论了共享文件夹备选方案。最终决定在设计中同时包含 Git 策略和兜底的共享文件夹方案。

---

### Q14：Git 策略细节

在概要设计中补充了完整的 Git 协作策略：
- 仓库初始化（P1 建骨架，打压缩包或 push 到远程）
- 分支命名（每人 `feature/xxx`）
- 两天时间线（何时 commit、何时合并）
- 合并步骤（P1 逐个合并，每次验证能跑）
- 冲突预防表（9 个可能冲突的文件，逐一说明预防方式）
- 兜底方案（不用 Git 就共享文件夹）

---

### Q15：Claude Code 启动 Prompt

**决定：** 为每个角色单独写一份可直接复制粘贴的 Claude Code 启动 Prompt，包含任务清单、依赖说明、可用接口、参考文件、约束条件。另建独立文档存放。

## 最终设计确认

转向方案 1 后重新确认了全部设计内容：

1. **需求不变** — 7 项需求全部保留
2. **技术架构** — FastAPI + 原生 HTML/CSS/JS MPA + JSON 文件存储（与工单系统一致）
3. **分工** — 4 人全栈 + 1 人测试/集成/文档/演示
4. **JSON 数据模型** — 6 个实体（users, templates, contracts, approval_records, messages, audit_logs）+ sessions
5. **合同状态机** — 9 状态线性流转（含驳回回退和续签）
6. **API 接口** — 20 个端点（与方案 3 时期一致）
7. **前端页面** — 11 个 MPA 页面，按角色分目录
8. **Git 协作策略** — 分支命名 + 合并步骤 + 冲突预防
9. **P5 演示职责** — 演示脚本 + 预制数据 + 周五主讲
10. **Claude Code 启动 Prompt** — 每人一份独立 Prompt 文档
