# 企业合同管理系统 — 概要设计

> 日期：2026-07-08 | 团队：5 人 | 周期：2 天开发 + 1 天汇报
> 架构方案：FastAPI + 原生 HTML/CSS/JS 多页面 + JSON 文件存储

---

## 一、项目概述

企业合同全生命周期管理系统，纯内部使用。支持合同拟制、两级审批（部门主管 → 法务）、签订归档、到期自动提醒、模板管理、全文检索。

参考项目：[企业售后工单系统](enterticketsystem/) — 复用其完整架构模式（FastAPI + MPA + JSON 持久化 + Session 认证），在此基础上适配合同管理业务。

---

## 二、需求摘要

| 编号 | 需求项 | 说明 |
|------|--------|------|
| R1 | 纯内部系统 | 仅企业内部人员使用，无外部角色 |
| R2 | 固定两级审批 | 部门主管 → 法务/财务，所有人走同一套流程 |
| R3 | 模板 + 自由填写 | 管理员预设模板（含可填字段定义），经办人也可空白创建 |
| R4 | 完整生命周期 | 草稿 → 审批中 → 驳回/通过 → 签订 → 履行中 → 到期/续签 → 已归档 |
| R5 | 全文检索 | 检索合同标题和对方名称（简单模糊匹配） |
| R6 | 到期自动提醒 | 系统自动扫描即将到期合同，给经办人发站内消息 |
| R7 | 三种角色 | 经办人（handler）、审批人（approver，含主管+法务）、管理员（admin） |

---

## 三、角色定义

| 角色 | 标识 | 权限范围 |
|------|------|----------|
| 经办人 | `handler` | 创建/编辑/提交/签订/归档自己的合同，查看关联合同，接收消息 |
| 审批人 | `approver` | 查看待审批合同，执行审批（通过/驳回），查看已审批记录，接收消息 |
| 管理员 | `admin` | 用户管理（创建/启用/禁用），模板管理，全局合同查看，数据看板 |

### 并行开发策略（不使用 Git，通过共享文件夹协作）

1. **第一天上午**：P1 创建项目骨架（[main.py](backend/app/main.py)、[config.py](backend/app/config.py)、JSON 存储层、路由注册占位），放入共享文件夹
2. **第一天上午同步**：P2/P3/P4 从共享文件夹拿到骨架，各自在独立子目录中开发（每个人只写自己的文件，不碰别人的）
3. **P5 全程并行**：从第一天起就写测试用例（mock 模式），同时完善需求分析和设计文档。P5 是唯一接触全部模块的人，周五由 P5 负责系统演示讲解
4. **第二天下午**：P1 负责集成——每个人把自己的文件放到共享文件夹对应位置，P1 验证整体可运行
5. **每人的文件独立，不存在合并冲突**：路由在 [main.py](backend/app/main.py) 中 P1 已预留注册，页面按角色分目录存放

### 模块间接口依赖

- P2（合同）→ P3（审批）：合同提交 → 调用 `approval_service.create_approval_records(contract_id)` → P3 负责
- P3（审批/消息）→ P1（认证）：读取 `get_current_user()` → P1 提供
- P4（模板）→ P2（合同）：合同创建时调用 `template_service.get_template(template_id)` → P4 提供
- 跨模块调用：在自己的 service 中 `from app.services.xxx_service import xxx`，因为文件是各人自建，引用时注意函数签名一致

### Git 协作策略

#### 仓库初始化（P1 负责）

```bash
# 在共享网盘或某人的电脑上
git init contract-system
cd contract-system
# P1 先建好项目骨架，然后：
git add .
git commit -m "feat: 项目骨架 — FastAPI + JSON存储 + 认证模块"
# 压缩打包，发到群里
zip -r contract-system-v1.zip contract-system/
```

全组拿到压缩包后各自解压，进入目录即可：

```bash
git checkout -b feature/xxx    # 各自建自己的分支
```

## 五、技术架构

```
浏览器（原生 HTML/CSS/JS 多页面应用）
        |
FastAPI REST API + 静态文件服务
        |
JSON 文件持久化（原子写入 + 线程锁）
```

| 层 | 技术选型 | 说明 |
|------|----------|------|
| 后端框架 | FastAPI (Python 3.10+) | 单进程运行 |
| 前端 | 原生 HTML / CSS / JavaScript | 无框架、无构建工具、ES Modules |
| 持久化 | JSON 单文件 | `backend/data/store.json`，原子写入 + 线程锁 |
| 认证 | 服务端 Session + HttpOnly Cookie | `SameSite=Lax`，8 小时 TTL |
| 密码 | Argon2id 哈希 | argon2-cffi 库 |
| 标识 | UUID v4 | 所有实体主键 |
| 时间戳 | UTC ISO 8601 (`Z` 后缀) | 统一格式 |
| 数据验证 | Pydantic v2 | 请求/响应模型 |

## 七、合同状态机

```
                  ┌──────────┐
                  │  draft   │ 草稿
                  └────┬─────┘
                       │ submit
                  ┌────▼─────┐
           ┌──────│  review  │ 审批中
           │      └────┬─────┘
           │ reject    │ approve (两级通过)
           │      ┌────▼─────┐
           └──────│ approved │ 已通过
                  └────┬─────┘
                       │ sign
                  ┌────▼─────┐
                  │  signed   │ 已签订
                  └────┬─────┘
                       │ (自动)
                  ┌────▼─────┐
                  │  active   │ 履行中
                  └──┬───┬───┘
              expire │   │ renew
                  ┌──▼─┐ ┌─▼──────┐
                  │expired││renewed │ 到期 / 续签
                  └──┬───┘ └──┬────┘
                     │ archive │ archive
                  ┌──▼──────▼──┐
                  │  archived  │ 已归档
                  └────────────┘
```

### 状态流转规则

| 当前状态 | 允许的下一个状态 | 触发操作 | 操作人 |
|----------|-----------------|----------|--------|
| draft | review | 提交审批 | 经办人 |
| review | approved | 两级审批均通过 | 审批人（主管→法务） |
| review | rejected | 任一审批人驳回 | 审批人 |
| rejected | review | 修改后重新提交审批 | 经办人 |
| approved | signed | 确认签订 | 经办人 |
| signed | active | 自动流转 | 系统 |
| active | expired | 到期（基于 expires_at 判定） | 系统 |
| active | renewed | 续签 | 经办人 |
| expired | archived | 归档 | 经办人/admin |
| renewed | archived | 归档 | 经办人/admin |

### 审批规则

- 合同提交审批后，系统按 `approver_1_id` → `approver_2_id` 顺序生成两条 `approval_records`（decision=pending）
- 一级审批人（主管）审批通过后，二级审批人（法务）才能审批
- 任一级驳回 → 合同状态回到 `rejected`，审批记录保留
- 重新提交 → 旧审批记录更新为 pending，从头开始

## 八、API 接口设计

所有接口前缀：`/api/v1`  
所有错误响应统一格式：`{"error": {"code": "...", "message": "中文描述"}}`

### 认证 `/api/v1/auth`

| 方法 | 路径 | 说明 | 权限 | 负责人 |
|------|------|------|------|--------|
| POST | `/register` | 创建账号 `{username, email, password, role}` | admin | P1 |
| POST | `/login` | 登录 `{username_or_email, password}`，返回 Set-Cookie | 公开 | P1 |
| POST | `/logout` | 退出，销毁会话 | 登录用户 | P1 |
| GET | `/me` | 当前用户信息 | 登录用户 | P1 |

### 合同 `/api/v1/contracts`

| 方法 | 路径 | 说明 | 权限 | 负责人 |
|------|------|------|------|--------|
| GET | `/` | 合同列表，`?status=&q=` 筛选和搜索 | handler 看自己的，approver 看待审批的，admin 看全部 | P2 |
| POST | `/` | 创建合同（draft）`{title, counterparty, amount?, template_id?, content_json, approver_1_id, approver_2_id, expires_at?}` | handler | P2 |
| GET | `/{id}` | 合同详情（含审批记录和操作日志） | 关联用户 | P2 |
| PUT | `/{id}` | 编辑合同（仅 draft/rejected） | 经办人 | P2 |
| POST | `/{id}/submit` | 提交审批（draft/rejected → review） | 经办人 | P2 |
| POST | `/{id}/sign` | 确认签订（approved → signed → active） | 经办人 | P2 |
| POST | `/{id}/archive` | 归档（expired/renewed → archived） | 经办人/admin | P2 |

### 审批 `/api/v1/contracts/{id}/approvals`

| 方法 | 路径 | 说明 | 权限 | 负责人 |
|------|------|------|------|--------|
| GET | `/` | 查看所有审批记录 | 关联用户 | P3 |
| POST | `/approve` | 审批通过（自动判断级别）`{comment?}` | 对应审批人 | P3 |
| POST | `/reject` | 审批驳回 `{comment?}` | 对应审批人 | P3 |

### 模板 `/api/v1/templates`

| 方法 | 路径 | 说明 | 权限 | 负责人 |
|------|------|------|------|--------|
| GET | `/` | 模板列表 | 登录用户 | P4 |
| POST | `/` | 创建模板 `{name, description, fields_json}` | admin | P4 |
| PUT | `/{id}` | 编辑模板 | admin | P4 |
| PATCH | `/{id}/status` | 启用/停用 `{is_active: bool}` | admin | P4 |

### 消息 `/api/v1/messages`

| 方法 | 路径 | 说明 | 权限 | 负责人 |
|------|------|------|------|--------|
| GET | `/` | 我的消息列表，`?is_read=` 筛选 | 登录用户 | P3 |
| GET | `/unread-count` | 未读消息数 | 登录用户 | P3 |
| PATCH | `/{id}/read` | 标记已读 | 消息接收人 | P3 |

### 管理员 `/api/v1/admin`

| 方法 | 路径 | 说明 | 权限 | 负责人 |
|------|------|------|------|--------|
| GET | `/users` | 用户列表 | admin | P4 |
| POST | `/users` | 创建用户 | admin | P4 |
| PATCH | `/users/{id}/status` | 启用/禁用 `{is_active: bool}` | admin | P4 |
| GET | `/stats` | 数据看板（合同总数/审批通过率/即将到期数/状态分布） | admin | P4 |

### 审批接口调用时序

```
经办人: POST /contracts              → status=draft
经办人: PUT  /contracts/{id}         → 编辑完善内容
经办人: POST /contracts/{id}/submit  → status=review, 创建两条审批记录(pending)
主管:   POST /contracts/{id}/approve → 一级通过
法务:   POST /contracts/{id}/approve → 二级通过, status=approved
经办人: POST /contracts/{id}/sign    → status=signed→active
```

### 错误码

| 错误码 | HTTP | 说明 |
|--------|------|------|
| `AUTHENTICATION_REQUIRED` | 401 | 未登录或会话失效 |
| `LOGIN_FAILED` | 401 | 账号/密码错误或账号被禁用 |
| `FORBIDDEN` | 403 | 角色权限不足 |
| `RESOURCE_NOT_FOUND` | 404 | 记录不存在 |
| `VALIDATION_ERROR` | 422 | 字段校验失败 |
| `CONFLICT` | 409 | 数据冲突（用户名/邮箱重复） |
| `INVALID_TRANSITION` | 409 | 不合法的状态流转 |
| `STORAGE_ERROR` | 500 | 持久化失败 |

### 全局交互规则

- 每个 HTML 页面内联 `<script type="module">` 加载对应 JS 模块
- 页面加载时调用 `GET /api/v1/auth/me`，401 则跳转登录页
- 导航根据 `user.role` 渲染不同菜单项（由 `session-ui.js` 负责）
- 前端按钮隐藏不构成安全措施，后端做最终权限校验
- 所有 API 请求通过 `api-client.js` 统一处理：`credentials: "include"` + 错误码映射

## 十、到期提醒机制

- 系统扫描所有 `status=active` 且 `expires_at` 不为空的合同
- 判断距到期时间 ≤ `EXPIRY_WARN_DAYS`（默认 30 天）
- 自动创建 `messages` 记录发送给合同经办人
- 触发时机：管理员手动触发 或 应用启动时执行一次
- MVP 范围内不做定时后台任务
